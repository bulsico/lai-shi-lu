# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
pnpm dev              # dev server on :3000
pnpm build            # prisma generate + next build
pnpm start            # production server on :3004
pnpm tsc --noEmit     # type-check (no test suite)
pnpm prisma:push      # apply schema changes to SQLite
pnpm prisma:studio    # browse DB

# Portable skill scripts
pnpm fetch-hl 0x...                              # fetch HL fills + open positions
pnpm fetch-hl 0x... --out data/tmp/fills.json
pnpm analyze --fills data/tmp/fills.json --out data/tmp/summary.json
```

No linter is configured. Type-check is the only automated correctness gate.

## Architecture

### Report generation pipeline

The core flow when a user submits an address:

1. **`POST /api/analyze`** — fetches fills + open positions from Hyperliquid concurrently, computes stats (`src/lib/analyze.ts`), writes a `data/tmp/{address}-summary.json`, then **spawns `claude -p` as a detached child process** running the `/trade-roast` skill
2. The skill reads the summary JSON and writes the final Markdown report to `data/reports/{address}.md`
3. The child process's `close` event updates the `Job` and `Report` rows in SQLite
4. **`GET /api/status/[sessionId]`** — polled by the frontend every 2.5s to check job completion
5. **`GET /api/report/[address]`** — returns the completed report from SQLite

Session state (PID, start time, exit code) lives in `data/tmp/sessions/{sessionId}.json` — separate from the DB, used to detect whether the Claude process is still alive.

### Key constraint: reports are generated by Claude Code CLI

`src/app/api/analyze/route.ts` hardcodes `CLAUDE_BIN = "/home/exedev/.local/bin/claude"` and uses `--dangerously-skip-permissions`. The process is detached (`claude.unref()`) so it outlives the HTTP request. Concurrency is capped at `MAX_CONCURRENT = 3` via an in-memory counter. Reports are cached for 24 hours.

### Data flow: two parallel codebases

The skill (`.claude/commands/trade-roast.md`) and scripts (`scripts/`) are a **portable version** that duplicates logic from `src/lib/`. Key duplication:
- `scripts/fetch-hl.ts` ≈ `src/lib/fetch-hl.ts` (standalone, no path aliases)
- `scripts/analyze.ts` ≈ `src/lib/analyze.ts` (standalone, with CLI entry point)

When changing fetch or analysis logic, **update both the `src/lib/` module and the corresponding `scripts/` file**.

### Frontend polling model

The report page (`src/app/report/[address]/page.tsx`) follows this logic on mount:
- If no `?session=` param → fetch report directly
- If `?session=` param → first check if report already exists (avoids re-showing loading screen on return); only start polling `/api/status` if the report isn't ready yet

### Database schema

Two tables in SQLite (`DATABASE_URL` env var, default `file:./dev.db`):
- `Report` — one row per address; `markdown` is empty string while generating, populated on completion
- `Job` — one row per analysis run; `status` is `pending` → `done` | `error`

### Skill prompt

`.claude/commands/trade-roast.md` defines the full report format that Claude follows when generating. It handles two data sources: Hyperliquid address (reads pre-computed summary JSON) and Robinhood CSV (`--rh` flag, runs `scripts/parse-rh.ts`). The summary JSON schema is defined by `TradeSummary` in `src/lib/analyze.ts`.
